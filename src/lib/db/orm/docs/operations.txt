If you're building a professional-grade ORM layer for PostgreSQL, you want names that are semantic (describing what they do) and consistent.

Since Postgres is a relational database, your naming convention should reflect common SQL operations but in a "developer-friendly" way. Here is the definitive list of operation names used by top-tier libraries (like Prisma, Drizzle, and TypeORM) and what they actually do in the background.

1. Retrieval Operations (Read)
These are the most common. In a professional ORM, you distinguish between finding one vs. many.

findUnique: Finds a single record using a unique identifier (like an ID or email). Uses a WHERE clause on a unique column.

findFirst: Finds the very first record that matches a criteria. Useful when you don't have a unique key but just want the "newest" or "top" result.

findMany: Returns an array of records. Usually supports where, orderBy, take (limit), and skip (offset).

count: Executes a SELECT COUNT(*) to return the total number of rows matching a filter.

aggregate: Used for math operations like sum, avg, min, or max on specific columns.

groupBy: Groups rows that have the same values into summary rows.

2. Mutation Operations (Write/Update)
Postgres handles these with INSERT and UPDATE commands.

create: Inserts a single new record.

createMany: Perfroms a bulk insert (highly efficient in Postgres using a single INSERT statement with multiple value sets).

update: Modifies a single record based on an ID.

updateMany: Modifies all records that match a specific criteria (e.g., "set active = false for all users who haven't logged in for a year").

upsert: A "Pro" favorite. Short for Update or Insert. In Postgres, this uses the ON CONFLICT clause. If the record exists, update it; if not, create it.

3. Removal Operations (Delete)
delete: Removes a single record.

deleteMany: Removes all records matching a filter. (Be careful with this one—it's the "danger zone" of API methods).

4. Advanced/Relationship Operations
Since you are using Postgres, you’ll eventually need to deal with linked tables.

include / join: Fetches related data (e.g., getting a User and their Posts in one go).

transaction: Not a table operation per se, but a way to wrap multiple operations (like create and update) so that if one fails, they all roll back.
